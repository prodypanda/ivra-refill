# Ivra Release Checklist

Use this checklist when moving from demo/local builds to real hotel pilots.

## Android Identity

- Package ID: `com.ivra.refill`
- App label: `Ivra Refill`
- Launcher icon: generated with `.\scripts\generate_android_icons.ps1`
- Adaptive icon: `mipmap-anydpi-v26/ic_launcher.xml`
- Launch splash: Ivra green background with generated `launch_image`
- Android 12+ splash: `values-v31/styles.xml`
- Debug APK command:

```powershell
.\scripts\build_android_debug.ps1
```

## Android Release Signing

Create a private keystore outside source control, then add `android/key.properties`:

```properties
storeFile=C:/secure/ivra-release.jks
storePassword=YOUR_STORE_PASSWORD
keyAlias=ivra
keyPassword=YOUR_KEY_PASSWORD
```

You can create both the keystore and `android/key.properties` with:

```powershell
.\scripts\setup_android_signing.ps1
```

Then build a release APK:

```powershell
.\scripts\build_android_release.ps1
```

Or build a release Android App Bundle:

```powershell
.\scripts\build_android_bundle.ps1
```

The project falls back to debug signing if `android/key.properties` is missing, which keeps local release-mode testing possible. Do not ship a debug-signed APK to hotels.
Production release build scripts require real Supabase values in `.env`. Use `-Demo` only when intentionally creating a non-production demo build.

## Web Release

- Confirm the Supabase production URL and anon key.
- Create the production `.env` with `.\scripts\setup_supabase_env.ps1`.
- Production web builds fail if `.env` does not contain real Supabase values. Use `.\scripts\build_web.ps1 -Demo` only for demo builds.
- Build web:

```powershell
.\scripts\build_web.ps1
```

- Deploy the contents of `build/web`.
- Use hash routes as generated by the app, for example `https://your-domain/#/rooms`.

## Supabase Release Gate

- Apply `supabase/migrations/0001_initial_schema.sql`.
- Run `supabase/bootstrap_first_admin.sql`.
- Create real role accounts.
- Run `supabase/rls_verification.sql`.
- Confirm refill, undo, correction, approval, inventory, alert, export, and invitation flows against live data.
- Record the production setup and pilot sign-off in `docs/08-go-live-record.md`.

## Final QA

- Run:

```powershell
.\scripts\verify_local.ps1 -BuildAndroid
```

- Run:

```powershell
.\scripts\check_release_readiness.ps1
```

- Build deployable artifacts:

```powershell
.\scripts\package_release.ps1 -IncludeAppBundle
```

- Keep `.generated/release/release-manifest.json` with the deployment handoff notes.
- Archive release artifacts for handoff:

```powershell
.\scripts\archive_release_artifacts.ps1 -IncludeAppBundle
```

- Verify the archive checksums:

```powershell
.\scripts\verify_release_archive.ps1
```

Or run the combined go-live preparation command:

```powershell
.\scripts\prepare_go_live.ps1 -IncludeAppBundle
```

- Keep `.generated/release/go-live-evidence.md` with release handoff notes.
- Test Chrome web on desktop and mobile width.
- Install the debug or release APK on an Android device.
- Confirm English, French, and Arabic UI.
- Confirm Arabic RTL layout.
- Complete `docs/08-go-live-record.md` without adding secrets.
